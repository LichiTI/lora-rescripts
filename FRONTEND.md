# 前端补丁规范

## 适用背景

当前仓库不提供可维护的前端源码，实际可编辑内容主要是 `frontend/dist` 发布产物。

这意味着前端变更必须以“可持续补丁”为目标，而不是临时把逻辑塞进编译主包。

## 允许的补丁结构

新增前端功能时，只允许使用以下三层结构：

1. 页面壳
   - 只负责在 HTML 页面壳中追加脚本引用。
   - 不在页面壳里写业务逻辑。
2. boot 层
   - 负责路由识别、页面识别、挂载、卸载、轮询启动与公共事件监听。
   - 文件命名建议使用 `*-boot.js`。
3. widget 层
   - 负责具体 UI、接口请求、状态刷新、用户交互。
   - 文件命名建议使用 `*-widget.js`。

## 禁止事项

- 禁止直接修改 `frontend/dist/assets/app.*.js`、`layout.*.js` 等编译主包，除非已经明确确认无法通过三层补丁实现。
- 禁止把业务逻辑直接写进页面壳内联脚本。
- 禁止为每个新功能各自发散一套完全不同的挂载方式。
- 禁止同一类运行时卡片重复实现轮询、重启、日志展示等近似逻辑而不抽象复用。

## 推荐做法

- 优先复用统一 boot 机制，避免一个功能一个入口。
- 页面壳改动应尽量保持机械化、最小化，通常只增加一个 `script type="module"`。
- widget 内只处理本功能本身，不承担路由分发职责。
- 新增运行时安装卡片时，优先对齐已有模式，例如：
  - `frontend/dist/assets/yolo-runtime-widget.js`
  - `frontend/dist/assets/aesthetic-runtime-widget.js`
  - `frontend/dist/assets/aesthetic-runtime-boot.js`

## 无法满足三层补丁时

如果需求确实无法通过 `页面壳 + boot + widget` 完成，必须先明确写出以下内容，再决定是否修改主包：

- 当前限制点是什么
- 为什么 boot 层无法接管
- 为什么 widget 无法独立挂载
- 为什么必须改编译主包

没有这一步，不应直接改 `app.*.js`。

## 提交前检查

- 是否避免了对 `app.*.js`、`layout.*.js` 的直接修改
- 是否把页面识别和挂载逻辑放在 boot 层
- 是否把 UI 与接口逻辑放在 widget 层
- 是否尽量复用了已有 runtime widget 的公共模式
- 是否把页面壳改动控制在最小范围
